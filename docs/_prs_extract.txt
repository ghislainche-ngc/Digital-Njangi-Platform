1. Introduction
1.1 Purpose
This Product Requirements Specification (PRS) defines the complete functional and non-functional requirements for NAAS (Njangi as a Service), a multi-tenant web platform designed to digitise the operation of Njangi rotating-savings groups in Cameroon. The document draws from the User Requirements Specification (URS) and the Project Brief, translating validated user needs into precise, implementable product requirements. It also articulates how Object-Oriented Programming (OOP) principles are applied throughout the system design.

1.2 Scope
NAAS is a Progressive Web Application (PWA) that allows any Njangi group in Cameroon to subscribe and manage contributions, payouts, fines, members, and communication from a single digital platform. The MVP scope includes group onboarding, role-based dashboards, MoMo payment collection and disbursement, automated scheduling, WhatsApp/SMS notifications, an immutable group ledger, and offline support. Features such as credit scoring, micro-loans, and bank integrations are explicitly out of scope for version 1.0.

1.3 Intended Audience
This document is intended for the NAAS development team (CTO, Dev A, Dev B), the Product Owner, the academic supervisor (Engr. TEKOH PALMA), and the defence panel. It serves as the engineering contract for what must be built.

2. System Overview
2.1 What We Are Building
NAAS is a multi-tenant SaaS platform. One codebase serves many independent Njangi groups, each fully isolated from one another at the data layer. When a Njangi subscribes, they receive a private workspace with member accounts, role-based dashboards, automated payment collection via MTN MoMo and Orange Money, a transparent immutable ledger, an automated payout engine, and WhatsApp notifications.

2.2 Architecture Summary
The system follows a multi-tenant monolith pattern: a single Node.js/Express backend, a PostgreSQL database managed by Supabase with Row-Level Security (RLS) enforcing tenant isolation, static HTML/Tailwind CSS frontend deployed as a PWA, and third-party integrations for payments and messaging. The monolith was deliberately chosen over microservices to match the team size and delivery timeline.

3. Object-Oriented Design Ś Rationale and Application
NAAS is built using Object-Oriented Programming (OOP) throughout its backend architecture. This section explains why OOP was chosen for this project, how the four pillars of OOP are applied, and what benefit each pillar delivers to the system.

3.1 Why OOP Was Chosen for NAAS
NAAS is not a simple CRUD application. It models a rich real-world domain Ś Njangi groups Ś that has natural entities (Group, Member, Contribution, Payout, Fine, Rotation), complex rules (eligibility checks, anti-fraud controls, rotation scheduling), and multiple actors with different permissions. OOP is the most natural fit because:
The domain contains distinct, well-bounded real-world entities that map cleanly to classes.
Business rules are tightly coupled to specific entities (e.g., a Payout knows its own eligibility rules).
Multiple payment providers (MTN MoMo, Orange Money) share an interface but differ in implementation Ś a textbook case for polymorphism.
The platform must support three rotation strategies that are interchangeable at runtime Ś ideal for the Strategy pattern via polymorphism.
Role-based access means every user shares common properties but has specialised behaviour Ś a direct use case for inheritance.
The payout engine, scheduler, and ledger are complex subsystems that must hide internal implementation from the rest of the code Ś encapsulation keeps the codebase maintainable for a four-person team.

3.2 The Four Pillars of OOP in NAAS

3.2.1 Encapsulation Ś Protecting Data and Hiding Complexity
Encapsulation means bundling data and the methods that operate on it inside a class, and exposing only what other parts of the system need to know. In NAAS, encapsulation is applied in two ways.
First, domain objects protect their own data integrity. The Contribution class, for example, holds its amount, status, timestamp, and member reference as private fields. External code cannot directly mutate these fields; it must call methods like markAsPaid() or markAsFailed(), which enforce business rules before making any change. This prevents a bug in the Treasurer dashboard from accidentally corrupting a Contribution record by bypassing validation.
Second, complex subsystems expose clean interfaces. The PayoutEngine class encapsulates the entire five-step payout flow Ś eligibility check, pot calculation, MoMo disbursement, ledger update, and WhatsApp notification Ś behind a single public method: execute(groupId, recipientId). The Treasurer dashboard calls this one method without needing to know anything about the internal steps. If the eligibility logic changes, only PayoutEngine needs to change, not the dashboard code.
Similarly, the MoMoPaymentService encapsulates all HTTP communication with the MTN API, token management, and retry logic. The rest of the codebase only calls charge(phoneNumber, amount) and receives a result.

3.2.2 Inheritance Ś Shared Structure, Specialised Behaviour
Inheritance allows a child class to inherit the properties and methods of a parent class while adding or overriding behaviour specific to its role. In NAAS, the user hierarchy is the primary application of inheritance.
All five user types Ś PlatformAdmin, President, Treasurer, Secretary, and Member Ś share a common User base class that holds: userId, email, phoneNumber, hashedPassword, role, createdAt, and methods like authenticate(), updateProfile(), and receiveNotification(). This shared implementation is written once in User and inherited by every subclass.
Each subclass then adds only the methods relevant to its role. President adds approvePayout(), inviteMember(), and configureRotation(). Treasurer adds recordCashPayment(), triggerMoMoPayout(), and applyFine(). Secretary adds postAnnouncement() and createMeetingMinutes(). Member adds viewLedger() and linkMoMoWallet(). PlatformAdmin adds suspendGroup() and viewPlatformAnalytics().
This design eliminates duplication. Authentication and notification logic are never repeated across five user types. When a bug is fixed in User.receiveNotification(), all five user types benefit automatically.
Importantly, President, Treasurer, and Secretary also inherit the Member subclass Ś because in a real Njangi, leaders are always members of their own group. The President class extends Member, which extends User. This reflects the actual domain accurately.

3.2.3 Polymorphism Ś One Interface, Many Implementations
Polymorphism allows different classes to be used interchangeably when they implement the same interface, and for the correct implementation to be selected at runtime without the caller needing to know which one is active.
NAAS applies polymorphism in two major areas.
The first is payment processing. Both MTNMoMoService and OrangeMoneyService implement a common PaymentProvider interface that defines: charge(phoneNumber, amount), disburse(phoneNumber, amount), getTransactionStatus(transactionId), and refund(transactionId). The PayoutEngine only ever calls methods on a PaymentProvider reference. At runtime, Supabase injects the correct implementation based on the member's registered wallet type. This means adding a third payment provider in the future requires only creating a new class that implements PaymentProvider Ś zero changes to PayoutEngine.
The second is rotation strategy. The project brief requires three rotation types: fixed schedule, random draw, and president decides. Each is implemented as a class (FixedRotationStrategy, RandomDrawStrategy, PresidentDecisionStrategy) that implements a common RotationStrategy interface with a single method: selectNextRecipient(memberList, history). The RotationEngine holds a reference to a RotationStrategy and calls selectNextRecipient() without knowing which strategy is active. The strategy is set once at group creation and injected at runtime. Switching rotation type between cycles requires only swapping the strategy object, not rewriting the rotation engine.

3.2.4 Abstraction Ś Modelling the Domain at the Right Level
Abstraction means representing real-world entities in the system at the level of detail needed to solve the problem Ś hiding irrelevant complexity and surfacing only the essential behaviour.
In NAAS, abstraction is applied through abstract base classes and interfaces that define contracts without committing to implementation.
The abstract PaymentProvider class defines what a payment service must be able to do Ś charge, disburse, check status, refund Ś without specifying how. This contract is the only thing the rest of the system depends on. The MTN MoMo and Orange Money implementations are free to differ in their HTTP calls, authentication schemes, and retry logic without any impact on the code that uses them.
The abstract NotificationService class defines the contract send(userId, message, channel) that all notification implementations must satisfy. WhatsAppNotificationService and SMSNotificationService implement this differently Ś one calls the Twilio WhatsApp Business API, the other calls an SMS gateway Ś but the PayoutEngine only depends on NotificationService. Adding an email channel means adding one new class with no changes to any caller.
Abstraction also applies to the RotationStrategy interface described in the polymorphism section. The interface abstracts the concept of 'choosing the next recipient' without encoding any particular algorithm.
This approach directly supports the team's ability to work in parallel: Dev A and Dev B can build different features that depend on the same abstract PaymentProvider or NotificationService interface, even before the concrete implementations are finished Ś because they only depend on the contract, not the implementation.

9. Definition of Done and Acceptance Criteria
9.1 Feature-Level Definition of Done
A feature is only considered complete when ALL of the following are true:
The feature works as described in the Trello card and maps to a labelled Product Requirement in this document.
It has been tested on a real browser Ś Chrome on Android Ś if it is a UI feature.
There are no console errors or broken API calls.
No hardcoded secrets appear anywhere in the code.
A Pull Request has been opened with title, description, test instructions, and screenshots for UI changes.
At least one other developer has reviewed and approved the PR.
The CTO has given final approval and merged the PR into main.
The corresponding Trello card has been moved to the Done column.

9.2 System-Level Acceptance Criteria
The following scenarios must pass end-to-end before the system is considered production-ready:
A new Njangi group can be created, configured, and populated with members entirely through the platform without any manual database intervention.
A contribution round fires automatically on the scheduled date. Every member with a linked MoMo wallet receives a debit. Each event is logged in the Ledger. Each member receives a WhatsApp confirmation.
All three rotation types Ś Fixed Schedule, Random Draw, and President Decides Ś produce a correct payout to the right member, trigger the eligibility checks, and update the Ledger and calendar.
A failed deduction is retried automatically, logged as failed if the retry also fails, and triggers an alert to the President and Treasurer.
Data from Group A is completely invisible to a logged-in user from Group B. An automated RLS test must confirm this.
The platform's key screens load correctly without internet access, serving data from the PWA cache.
A Treasurer attempting to record a cash payout to themselves triggers a fraud alert to the President.
